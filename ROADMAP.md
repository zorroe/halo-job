# Halo-Job 迭代路线图

本文档用于指导 `halo-job` 后续迭代开发，目标不是一次性把所有能力做满，而是按照“先打地基、再补运行时、再做扩展平台、最后做治理能力”的顺序推进。

## 1. 当前基线

当前仓库已经具备以下能力：

- `Admin + Executor` 基本架构
- MySQL 持久化任务定义、执行器信息、执行日志
- Redis 分布式锁保护定时扫描
- Cron 自动调度
- 手动触发
- 执行器自动注册与心跳
- 路由策略：轮询、随机、首个、末尾、Hash、分片广播
- 阻塞策略：排队等待、丢弃新任务、覆盖运行中任务
- 简单失败重试

当前主要短板：

- 任务与执行器之间缺少“分组 / 应用 / handler 能力”绑定
- 触发模型单一，只有 Cron 表达式
- 缺少 Misfire 补偿策略
- 缺少任务实例模型，运行态信息和重试信息混在执行日志里
- 缺少超时控制、取消运行、重跑、分页查询等运行时能力
- 路由、阻塞、重试策略还不是可插拔架构
- 参数模型过于简单，只支持字符串
- 缺少观测、告警、权限、审计等平台能力

## 2. 迭代原则

- 优先保证模型正确性，再补功能丰富性
- 先做后端能力闭环，再考虑管理端体验
- 每个里程碑都要求“可发布、可验证、可回滚”
- 每次数据库变更都提供独立 SQL 升级脚本
- 每次能力扩展都尽量保留向后兼容，避免一次性推翻现有调用方式

## 3. 里程碑总览

| 版本 | 目标 | 关键词 |
| --- | --- | --- |
| `v0.2` | 打好调度模型地基 | 执行器分组、handler 注册、触发模型升级 |
| `v0.3` | 补齐运行时能力 | Misfire、任务实例、超时、取消、重跑 |
| `v0.4` | 建立扩展架构 | SPI、JSON 参数、元数据上报、策略插件化 |
| `v1.0` | 形成平台化能力 | 编排、告警、权限、审计、治理 |

建议开发节奏：

1. 先完成 `v0.2`
2. 再完成 `v0.3`
3. `v0.4` 期间逐步整理代码结构，降低后续扩展成本
4. `v1.0` 再进入平台治理和工作流能力

## 4. v0.2 调度模型地基

### 4.1 目标

解决“任务到底该发给谁、按什么规则触发、任务定义如何表达”的基础问题，为后续所有能力提供统一模型。

### 4.2 范围

- 执行器分组与应用维度
- handler 注册中心
- 任务定义模型升级
- 触发模型抽象
- 基础接口兼容改造

### 4.3 主要改动

#### 数据库

建议新增或调整以下字段。

`job_info`

| 字段 | 说明 |
| --- | --- |
| `executor_group` | 执行器分组，任务只会路由到该组 |
| `executor_app` | 执行器应用名，可选，用于进一步过滤 |
| `trigger_type` | 触发类型，先支持 `CRON`、`ONCE`、`FIXED_RATE`、`FIXED_DELAY` |
| `trigger_config` | 触发配置 JSON，替代只靠单列 `cron_expression` |
| `misfire_policy` | 错过触发时的策略 |
| `timeout_seconds` | 单次运行超时时间 |
| `max_concurrency` | 任务维度并发限制，先预留 |
| `owner` | 负责人 |
| `tag` | 标签，先做简单字符串或 JSON |

`executor_info`

| 字段 | 说明 |
| --- | --- |
| `executor_group` | 执行器分组 |
| `executor_app` | 执行器应用名 |
| `metadata` | 扩展元数据 JSON |
| `version` | 执行器版本 |

建议新增表：

`executor_handler`

| 字段 | 说明 |
| --- | --- |
| `executor_name` | 执行器名 |
| `executor_address` | 执行器地址 |
| `handler_name` | handler 名称 |
| `handler_desc` | handler 描述 |
| `method_signature` | 方法签名摘要 |
| `update_time` | 最近上报时间 |

#### 后端模型

- 执行器注册时，不只上报 `name/address`，还上报 `group/app/handlers/metadata`
- 调度中心查询在线执行器时，按 `executor_group + executor_app + handler` 过滤
- `job_info` 不再把 `cron_expression` 作为唯一调度配置来源
- 增加统一的 `TriggerDefinition` / `TriggerParser` 抽象

#### 接口

建议保留现有接口，同时新增更标准的管理接口。

- `POST /executor/api/register` 改为支持 JSON 上报
- `POST /admin/job/save` 支持新的任务定义字段
- 新增 `GET /admin/job/handlers`，用于查看某组可用 handler
- 新增 `GET /admin/executor/list`，支持按组、应用、状态过滤

### 4.4 里程碑验收标准

- 任务只能调度到匹配分组且支持目标 handler 的执行器
- 现有 Cron 任务能平滑迁移到新模型
- 执行器重启后能自动恢复 handler 注册信息
- 管理接口能查到在线执行器及其可执行 handler 列表
- 旧版 `/executor/run` 兼容调用不被破坏

### 4.5 本阶段不做

- DAG 编排
- 告警通知
- 权限体系
- 前端页面

## 5. v0.3 运行时能力补齐

### 5.1 目标

解决“任务触发后如何可靠运行、如何追踪一次任务、如何处理异常和超时”的问题。

### 5.2 范围

- Misfire 处理
- 任务实例模型
- 超时与取消
- 重试策略增强
- 日志与查询能力增强

### 5.3 主要改动

#### 数据库

建议新增表：

`job_instance`

| 字段 | 说明 |
| --- | --- |
| `id` | 实例 ID |
| `job_id` | 任务 ID |
| `trigger_type` | 触发来源 |
| `scheduled_time` | 原计划触发时间 |
| `actual_trigger_time` | 实际触发时间 |
| `status` | `WAITING/RUNNING/SUCCESS/FAILED/CANCELED/TIMEOUT` |
| `dispatch_count` | 下发次数 |
| `final_result` | 最终结果摘要 |
| `create_time` | 创建时间 |
| `update_time` | 更新时间 |

`job_attempt`

| 字段 | 说明 |
| --- | --- |
| `id` | 尝试 ID |
| `instance_id` | 对应实例 |
| `executor_address` | 实际执行节点 |
| `shard_index` | 分片序号 |
| `shard_total` | 分片总数 |
| `attempt_no` | 第几次尝试 |
| `status` | 尝试状态 |
| `error_msg` | 错误信息 |
| `start_time` | 开始时间 |
| `end_time` | 结束时间 |
| `cost_time` | 耗时 |

`job_execution_log` 可逐步退化为兼容视图或历史表，不再承载全部运行语义。

#### 调度逻辑

- 引入 Misfire 策略：
  - `DO_NOTHING`
  - `FIRE_ONCE_NOW`
  - `CATCH_UP_LIMITED`
- 调度中心启动时补扫“错过但仍需补偿”的任务
- 调度中心触发任务时先创建 `job_instance`
- 每次下发和重试都记录 `job_attempt`

#### 执行控制

- 增加任务超时控制
- 增加“取消运行中任务”接口
- 增加“按实例重跑”接口
- 阻塞策略继续沿用，但运行结果要纳入实例状态流转

#### 查询能力

- 日志改成分页查询
- 支持按任务、状态、时间范围、执行器过滤
- 支持查询实例详情、尝试详情

### 5.4 里程碑验收标准

- 能明确区分一次任务实例与多次重试尝试
- 超时任务能被标记为 `TIMEOUT`
- 取消任务后运行状态能准确反映到实例和尝试
- 管理接口支持分页和条件过滤
- 调度中心重启后能按 Misfire 策略正确补偿

### 5.5 本阶段不做

- 多节点 admin leader 选举
- 复杂工作流
- 插件市场

## 6. v0.4 扩展架构建设

### 6.1 目标

解决“每增加一种策略或任务类型都要改核心代码”的扩展性问题，把系统从硬编码实现演进为可插拔平台。

### 6.2 范围

- 策略 SPI
- 参数模型升级
- 执行器元数据上报
- 多任务类型扩展
- 核心包结构整理

### 6.3 主要改动

#### SPI 化改造

将以下能力从枚举 / `switch` 改造成接口 + 默认实现：

- 路由策略
- 阻塞策略
- 重试策略
- Misfire 策略
- 触发类型解析器
- 执行结果聚合器

建议定义接口：

- `RouteStrategy`
- `BlockStrategy`
- `RetryPolicy`
- `MisfirePolicy`
- `TriggerTypeHandler`
- `ExecutionResultAggregator`

#### 参数模型升级

- `executor_param` 从纯字符串升级为 JSON 文本
- 增加 `param_schema`
- 执行器支持把 JSON 参数映射为 `String`、`Map` 或后续扩展对象

第一阶段先支持：

- 继续兼容旧字符串参数
- 新任务默认走 JSON 参数

#### 元数据上报

增强 `@HaloJob` 注解，支持：

- `value`
- `desc`
- `timeoutSeconds`
- `defaultRetryCount`
- `tags`
- `paramSchemaRef`

执行器启动时把这些元数据同步到 admin。

#### 多任务类型

在不破坏现有 Java handler 的前提下，预留任务类型：

- `JAVA_HANDLER`
- `HTTP_CALLBACK`
- `SHELL_SCRIPT`

本阶段至少落地 `JAVA_HANDLER` 和 `HTTP_CALLBACK` 两种。

### 6.4 代码结构调整建议

- `halo-job-core` 继续保留通用契约
- 新增 `halo-job-spi`，放可插拔接口
- 新增 `halo-job-admin-domain` 或按包拆分，弱化 controller/service/mapper 直接耦合
- 后续若任务类型继续增多，可再拆 `executor-sdk`

### 6.5 里程碑验收标准

- 新增一个路由策略不需要修改核心 `switch`
- 新增一种触发类型可以通过实现接口接入
- 新任务可使用 JSON 参数配置
- 执行器上报的 handler 元数据可在 admin 侧查询
- HTTP 类型任务可通过统一调度入口触发

## 7. v1.0 平台化能力

### 7.1 目标

从“可运行的调度器”升级为“可治理的任务平台”。

### 7.2 范围

- 工作流编排
- 告警通知
- 权限认证
- 审计与发布
- 指标监控
- 平台治理能力

### 7.3 主要改动

#### 工作流编排

- 支持前置依赖
- 支持 DAG
- 支持失败分支和补偿任务
- 支持手动跳过节点、重试节点

#### 告警通知

- 任务失败告警
- 连续失败阈值告警
- 超时告警
- 执行器离线告警
- 告警通道先支持 Webhook

#### 权限与审计

- 登录认证
- 项目 / 分组级权限
- 任务变更审计
- 操作日志
- 发布历史与回滚记录

#### 指标与监控

- 成功率
- 平均耗时
- P95 / P99 耗时
- 积压任务数
- 在线执行器数
- 超时数
- 重试数

#### 治理能力

- 执行器摘除 / 隔离
- 任务限流
- 任务熔断
- 手工 reroute
- 租户隔离

### 7.4 里程碑验收标准

- 复杂任务链路可通过依赖关系编排执行
- 告警、审计、权限具备基本平台可用性
- 核心运行指标可查询可展示
- 出现执行器异常时，平台具备基本止损能力

## 8. 推荐开发顺序

建议严格按下面顺序开发，避免返工。

### 第 1 批：数据模型和接口地基

- 升级 `job_info`
- 升级 `executor_info`
- 新增 `executor_handler`
- 改造执行器注册协议
- 改造任务保存接口

### 第 2 批：调度路由正确性

- handler 注册和查询
- 执行器分组过滤
- 任务与执行器匹配校验
- 新触发模型落地

### 第 3 批：运行时模型

- `job_instance`
- `job_attempt`
- Misfire 扫描
- 分页日志查询

### 第 4 批：执行控制

- 超时
- 取消
- 重跑
- 重试策略增强

### 第 5 批：扩展架构

- SPI 抽象
- JSON 参数
- handler 元数据上报
- HTTP 任务类型

### 第 6 批：平台治理

- 告警
- 权限
- 审计
- 编排

## 9. 每个版本的交付要求

每个迭代版本都必须同时交付以下内容：

- 代码
- SQL 升级脚本
- README / ROADMAP 更新
- 接口文档
- 最小集成测试
- 兼容性说明

## 10. 下一步建议

后续开发请从 `v0.2` 开始，按以下顺序拆任务：

1. 先做数据库升级脚本和实体模型调整
2. 再做执行器注册协议升级和 handler 上报
3. 再做任务保存 / 查询接口改造
4. 最后改调度路由逻辑

如果后续按照这个路线图推进，推荐把 `v0.2` 再拆成 4 个小迭代：

- `v0.2.1` 执行器分组与注册协议
- `v0.2.2` handler 注册中心
- `v0.2.3` 任务定义模型升级
- `v0.2.4` 调度匹配与兼容性收口
